# Roadmap: Agent-Native Information Management System

> 核心原则：提升 agent 检索事实信息的精度和效率。智能在 agent 端，不在工具端。

## 第零阶段：当前状态（已完成）

- CLI 核心 + JSON 信封协议
- 订阅、存储、全文抓取
- 基于事实的文本特征检测（字数、有无代码、有无步骤等）
- 智能文件夹 = 事实查询视图
- Solid.js GUI 壳 + 键盘优先
- compact/browse/deep 三层 agent 输出

## 第一阶段：可靠的情报网络

**目标**：agent 可以信任这个系统作为稳定的信息源。

- [x] **Feed 健康监控** — 记录抓取失败率、响应时间、更新频率。发现"死掉的线人"。`feed-health` 命令。
- [ ] **去重** — 同一篇文章从多个 feed 进来（转载、聚合器），agent 不应重复处理相同信息。
- [x] **增量同步** — ETag/Last-Modified 条件请求，304 跳过，省带宽。
- [ ] **OPML 导入导出完善** — agent 需要程序化管理订阅列表。

## 第二阶段：更精准的检索

**目标**：从"给我文章"到"给我恰好需要的东西"。

- [ ] **结构化查询语言** — 组合事实谓词：`--where "has_code AND word_count > 2000 AND feed_id IN (1,3,5) AND published_after 2024-01-01"`。让 agent 自己组合查询，而不是只用预设智能文件夹。
- [x] **全文搜索（SQLite FTS5）** — BM25 相关性排序（标题 10x 权重），支持布尔查询/短语匹配/前缀搜索。`--sort relevance|date`。
- [ ] **跨 feed 实体关联** — `entities` 表已存在但未充分利用。"所有提到实体 X 的文章"是强力 agent 查询。
- [ ] **时间序列感知** — "过去 24 小时我的 feed 有什么变化？"作为一等查询。agent 按轮询周期工作。

## 第三阶段：说 Agent 的语言

**目标**：不只是 CLI 可调用，而是与 agent 生态原生互通。

- [ ] **MCP Server** — 把 feed/文章/搜索暴露为 MCP resource 和 tool。任何 MCP 兼容的 agent 直接接入。
- [ ] **新文章回调/webhook** — 推送模式，不只是轮询。agent 注册兴趣，系统主动通知。
- [ ] **批量操作** — `mark-read [id1,id2,...]`、`star [ids]`、`fetch-full-text [ids]`。agent 批量工作，不是一条一条。
- [ ] **Token 预算提示** — `--max-tokens 4000` 截断输出以适配 agent 上下文窗口。

## 第四阶段：信息拓扑

**目标**：从扁平列表到知识图谱结构。系统理解信息的**结构**，不是内容的含义。

- [ ] **Feed 关系图谱** — 哪些 feed 覆盖相似领域（基于实体共现，不是 AI 分类）。agent 可以发现"读了 A 源，B 源覆盖同一领域"。
- [ ] **文章线索追踪** — 检测后续报道、更正、同一事件的更新。靠事实：相同实体 + 时间接近 + URL 互引。
- [ ] **参与度加权排序** — 用 `read_rate`、`star_rate` 排序搜索结果。不是分类——只是"这个源历史上更有用"。
- [ ] **新鲜度衰减** — 文章随时间降低检索优先级，除非被标星或引用。保持索引的相关性。
- [ ] **向量语义搜索（数据驱动）** — 文章量积累到数千篇、FTS5 出现"词不达意"时引入。嵌入是检索能力的升级，不是分类——agent 说"找相似的"，系统返回结果，判断权仍在 agent。触发条件：出现"语义相关但关键词不重叠"的真实检索失败 case。
- [ ] **LLM 重排序（可选层）** — `search "query" --rerank`。BM25 先出候选集，LLM 按语义相关度重排。不改变默认行为，按需启用。和 BM25 调权重是同一性质——优化检索精度，不是替用户判断。参考 [tobi/qmd](https://github.com/tobi/qmd) 的三层架构：`search`（BM25）→ `vsearch`（向量）→ `query`（混合 + 重排序）。

## 第五阶段：多 Agent 协作

**目标**：不同目的的多个 agent 共享同一套信息基础设施。

- [ ] **命名 agent 会话/配置** — 不同 agent 有不同的订阅、已读状态、标星模式。一个"研究 agent"和一个"新闻监控 agent"共存。
- [ ] **注解层** — agent 可以给文章附加结构化元数据（键值对，不是自由文本 AI 标签）。其他 agent 可以查询这些注解。
- [ ] **导出管线** — `export --format jsonl --query "..."` 输出到外部系统。RSS 系统是信息源头，不是终点。

---

## 设计红线

**永远不做：**
- AI/LLM 文章**分类或归类**（系统不能替用户判断"这篇文章属于什么"）
- 推荐引擎（"你可能喜欢……"）
- 社交功能
- 显式反馈收集 UI
- 优化阅读体验（它是壳，不是阅读器）
- 任何"系统猜测用户想要什么"的功能

**可以做，但要等数据和需求驱动：**
- 向量嵌入/语义搜索/LLM 重排序 — 这些是**检索能力的升级**，不是判断。和 BM25 调权重同一性质。等数据量和真实需求驱动，作为可选层（`--rerank`）引入，不改变默认行为。

## 设计隐喻

造望远镜（更好的光学、更好的支架、更好的追踪），不是造机器人天文学家。这个项目是望远镜。
